## Getting Started

### Prerequisites

- **Go 1.25+** (see `go.work`)
- **Linux** recommended (control socket uses `SO_PEERCRED`, `mlock` for key protection)
- **OpenSSL** and **OpenSSH** (for `make testdata`)
- **Docker** (for E2E tests)

### Setup

```bash
git clone https://github.com/ba0f3/luna-ztrust.git
cd luna-ztrust

# Sync Go workspaces
go work sync

# Generate test certificates and keys
make testdata

# Run tests
make test

# Build binaries
make build
```

### Project Structure

```
luna-ztrust/
├── go.work              # Go workspace (multi-module)
```

## Development Workflow

### Dependency Rules (Strict)

```
agent  →  sdk
proxy  ✗  sdk
sdk    ✗  agent, proxy
```

- **Single protocol implementation**: HTTP, PoP, HMAC, mTLS live in `sdk` only
- **Proxy isolation**: `proxy` talks to Telegram and local signing; never imports `sdk`

### Making Changes

1. **Fork** the repository
2. **Create a feature branch** from `main`: `git checkout -b feat/your-feature`
3. **Make minimal, focused changes** — match existing naming and package layout
4. **Run tests locally**: `make ci` (fmt-check, lint, test, build)
5. **Update documentation** if behavior, flags, or APIs change:
   - `README.md` — user-facing behavior, env vars, API
   - `AGENTS.md` — architecture for contributors
   - Affected spec in `docs/superpowers/specs/`
   - Affected plan in `docs/superpowers/plans/`

### Commit Messages

Follow conventional commits:

```
feat(proxy): add CLI remote key load for local-key mode
fix(agent): handle socket path resolution on non-Linux
docs: update deploy.md with systemd user creation
```

## Code Style

- **Go**: Match standard library and `golang.org/x/crypto/ssh` patterns
- **Imports**: Top of file only; no inline imports
- **Linux-only code**: Use `//go:build linux` for `SO_PEERCRED` and platform-specific control socket behavior
- **Error handling**: Exhaustive handling for status/enum unions; use `fmt.Errorf` with `%w` for wrapping
- **English** for all code and comments
- **No AI slop**: Code should be indistinguishable from a senior engineer's

### Linting & Formatting

```bash
make fmt-check   # go fmt ./...
make lint        # go vet ./...
```

## Testing

### Test Layers

1. **Unit** — PoP, HMAC, LRU, tx FSM, keystore, control ops
2. **Integration** — mTLS API, CLI enroll/load, Telegram/mobile
3. **E2E** — docker-compose (see `deploy/docker-compose.e2e.yml`)

### Running Tests

```bash
# All unit + integration tests
make test

# E2E tests (requires Docker)
make e2e-up
make e2e-wait
make e2e-test
make e2e-down

# Single package
go test ./proxy/internal/keystore/... -v

# With race detector
go test -race ./...
```

### Test Data

Test certificates and SSH keys are generated by `make testdata` and stored in `testdata/` (gitignored). Passphrase for test keys: `test-pass`.

## Pull Request Process

1. **Ensure CI passes** — `make ci` must succeed
2. **Keep PRs focused** — one logical change per PR
3. **Update docs** — if you change behavior, flags, or APIs
4. **Add tests** for new functionality
5. **Reference issues** — `Fixes #123` or `Refs #123` in PR description

### PR Review Criteria

- Spec compliance (does code match the design?)
- Code quality (standards, security, performance, edge cases)
- Adversarial review (security holes, false assumptions, resource exhaustion, race conditions)
- Tests pass and coverage is maintained

## Security

### Reporting Vulnerabilities

**Do not file public issues for security vulnerabilities.** See [SECURITY.md](SECURITY.md) for the responsible disclosure process.

### Security Principles (Non-Negotiable)

- **Fail-closed** on all auth, key-load, and signing errors
- **Sealed keystore**: `POST /api/v1/ssh/sign` returns `503` until signer loaded
- **Never log**: private keys, TLS exporter keys, passphrases, encrypted PEM bodies, raw signatures
- **Source address**: from mTLS listener `RemoteAddr`; never trust `X-Forwarded-For`
- **Control plane**: Unix socket with `SO_PEERCRED` and configured admin group

### Known Limitations

See [Security Review 2026-06-06](docs/security-review-2026-06-06.md) for documented Critical/High/Medium findings that are being addressed.

## Architecture Overview

### Core Protocol (Approach 2)

```
POST /api/v1/ssh/sign → 202 + tx_id → GET /api/v1/ssh/sign/{tx_id}/wait → cert/signature
```

Optional `?wait=1` on POST for one round-trip; server still uses internal `tx_id`.

### Auth Pipeline Order (Proxy)

1. mTLS — reject missing/invalid client cert
2. HMAC — constant-time `X-Luna-Body-Mac` (TLS exporter label `luna-request-hmac`, 32 bytes)
3. Timestamp — ±30s
4. Replay LRU — `SHA256(raw_body)`, 60s TTL → `409` on duplicate
5. PoP — verify `pop_signature` against `public_key`
6. Create `tx_id` (ULID), enqueue approval

Auth failure → **no** `tx_id`, **no** Telegram.

### Modules

| Module | Role | Public API |
|--------|------|------------|
| `sdk` | Ephemeral keys, PoP, mTLS+HMAC, cert/signature HTTP client | `NewClient`, `RequestCertificate`, `RequestSignature`, `FetchCapabilities`, `NewCertSigner` |
| `proxy` | Central gateway: auth, control socket, keystore, signing, Telegram, leases | CLI only; HTTP API for agents/CLI/mobile |
| `agent` | `SSH_AUTH_SOCK` daemon; blocking `Sign` via SDK | `luna-agent` binary; implements ssh-agent protocol |

### Deferred (Not in v1)

- Kubernetes manifests
- FCM push (Telegram is v1)
- `memfd_create` subprocess key FD passing
- Global API keys (use mTLS + PoP + HMAC)
- Disk-persisted SSH private keys or Vault tokens on proxy
- Persistent CLI/mobile registries and keystore pools (v1 is in-memory)
